# P5 — Tool Use 코드 진화 가이드

이 문서는 P5 시리즈의 코드가 파일을 거치면서 어떻게 단순해지고 강력해지는지,  
그 진화 과정을 단계별로 설명합니다.  
코드를 처음 보기 전에 이 문서를 읽으면 "왜 이렇게 만들었는지"가 보입니다.

---

## 전체 진화 한눈에 보기

```
P5_02  →  P5_03  →  P5_04  →  P5_05  →  P5_06  →  P6_08
흐름이해   도구정의   함수포장   반복루프   대화기억   @tool 자동화
(날것)    (구조학습)  (재사용)   (다중도구)  (히스토리)  (프레임워크)
```

---

## 1단계 — P5_02: 4단계 흐름을 날것으로 이해

**목적:** Tool Use가 내부에서 어떻게 돌아가는지 뼈대를 보여주는 단계입니다.  
모든 것을 직접 작성하고, 단계마다 무슨 일이 벌어지는지 출력해서 확인합니다.

### 흐름

```
[Step 1] converse(질문 + toolConfig)
         → 모델: "get_weather 도구를 써야겠다"
         → stopReason = "tool_use"

[Step 2] 응답에서 toolUse 블록 꺼내기
         → 도구명: get_weather
         → 입력값: {"city": "Seoul"}
         → toolUseId: "tooluse_abc123"   ← 이 ID가 핵심!

[Step 3] 우리 코드가 실제 함수 실행
         → get_weather("Seoul") → {"report": "Seoul: ☀️ 29°C"}
         → toolResult 형식으로 조립

[Step 4] toolResult 담아 converse 재호출
         → 모델: "서울 날씨는 맑고 29°C입니다." (최종 답변)
```

### 핵심 코드

```python
# ① 함수 구현
def get_weather(city: str) -> dict:
    url = f"https://wttr.in/{city}?format=3"
    ...
    return {"city": city, "report": report}

# ② toolSpec — 모델에게 도구를 설명하는 JSON
WEATHER_TOOL = {
    "toolSpec": {
        "name": "get_weather",
        "description": "도시의 현재 날씨를 조회합니다.",
        "inputSchema": {
            "json": {
                "type": "object",
                "properties": {
                    "city": {"type": "string", "description": "도시명 (영문)"}
                },
                "required": ["city"],
            }
        },
    }
}
TOOL_CONFIG = {"tools": [WEATHER_TOOL]}

# ③ Step 1: 첫 번째 converse
resp = bedrock.converse(
    modelId=MODEL_ID,
    messages=[{"role": "user", "content": [{"text": "서울 날씨 알려줘"}]}],
    toolConfig=TOOL_CONFIG,
)

# ④ Step 2: toolUse 블록 파싱
for block in resp["output"]["message"]["content"]:
    if "toolUse" in block:
        tu = block["toolUse"]   # name, input, toolUseId

# ⑤ Step 3: 실제 함수 실행 → toolResult 조립
result = get_weather(**tu["input"])
tool_result_msg = {
    "role": "user",
    "content": [{
        "toolResult": {
            "toolUseId": tu["toolUseId"],   # ← Step 2의 ID와 반드시 일치!
            "content": [{"json": result}],
        }
    }]
}

# ⑥ Step 4: toolResult 담아 재호출
final = bedrock.converse(modelId=MODEL_ID, messages=[...+ tool_result_msg], toolConfig=TOOL_CONFIG)
```

### 주의사항

```
⚠️  toolUseId 불일치  → ValidationException (가장 흔한 오류)
⚠️  role 순서 위반    → "roles must alternate" 오류
    올바른 순서: user → assistant → user(toolResult)
⚠️  content 형식      → {"json": ...} 감싸기 필수
```

> 이 단계의 코드량: 약 **80줄**. 도구 1개, 호출 1번만 처리 가능.

## 2단계 — P5_03: 도구 정의 구조 학습

**목적:** toolSpec의 3요소를 명확히 이해합니다.

```
toolSpec 3요소:
┌─────────────────────────────────────────────────┐
│  name        → 모델이 호출할 때 쓰는 함수 이름   │
│  description → 언제/왜 쓰는지 설명 (매우 중요!)  │
│               description이 불명확하면 모델이    │
│               도구를 안 쓰고 그냥 답해버림        │
│  inputSchema → 파라미터 구조 (JSON Schema 형식)  │
└─────────────────────────────────────────────────┘
```

> P5_03은 구조 학습용입니다. toolResult 반환까지는 구현하지 않습니다.

## 3단계 — P5_04: `ask_with_tools()` — 4단계를 함수 하나로 포장

**목적:** P5_02의 흩어진 4단계를 **재사용 가능한 함수**로 캡슐화합니다.

### 추가된 것: `run_tool()` + `ask_with_tools()`

```python
# run_tool(): toolUse 블록 → toolResult 변환 (반복 작업 함수화)
TOOL_FUNCTIONS = {"get_weather": get_weather}  # 함수 등록표

def run_tool(tool_use: dict) -> dict:
    func   = TOOL_FUNCTIONS[tool_use["name"]]
    result = func(**tool_use["input"])
    return {
        "toolResult": {
            "toolUseId": tool_use["toolUseId"],
            "content": [{"json": result}],
        }
    }

# ask_with_tools(): 4단계 전체를 1번 호출로 처리
def ask_with_tools(user_text: str) -> str:
    messages = [{"role": "user", "content": [{"text": user_text}]}]

    resp = bedrock.converse(modelId=MODEL_ID, messages=messages, toolConfig=TOOL_CONFIG)

    if resp["stopReason"] != "tool_use":          # 도구 불필요 → 바로 반환
        return resp["output"]["message"]["content"][0]["text"]

    assistant_msg = resp["output"]["message"]
    messages.append(assistant_msg)

    tool_results = [run_tool(b["toolUse"]) for b in assistant_msg["content"] if "toolUse" in b]
    messages.append({"role": "user", "content": tool_results})

    final = bedrock.converse(modelId=MODEL_ID, messages=messages, toolConfig=TOOL_CONFIG)
    return final["output"]["message"]["content"][0]["text"]
```

### P5_02 대비 개선점

```
P5_02: 80줄, 매번 직접 작성
P5_04: ask_with_tools("서울 날씨 알려줘") ← 한 줄로 호출 가능

⚠️ 아직 한계:
   도구를 1번만 호출하는 경우만 처리
   모델이 도구를 2번 연속 쓰면 두 번째는 처리 못함
```

## 4단계 — P5_05: `run_conversation()` + `toolChoice` — 다중 도구 + 반복 루프

**목적:** 도구가 2개(`get_weather` + `get_time`)로 늘고,  
모델이 도구를 **여러 번 연속 호출**하는 경우도 처리합니다.

### 추가된 것 1: `run_conversation()` — 루프 방식

```python
TOOL_FUNCTIONS = {"get_weather": get_weather, "get_time": get_time}
TOOL_CONFIG    = {"tools": [WEATHER_TOOL, TIME_TOOL]}

def run_conversation(messages: list, max_turns: int = 5) -> str:
    for _ in range(max_turns):          # 최대 5번 루프 (무한루프 방지)
        resp = bedrock.converse(modelId=MODEL_ID, messages=messages, toolConfig=TOOL_CONFIG)
        assistant_msg = resp["output"]["message"]
        messages.append(assistant_msg)

        if resp["stopReason"] != "tool_use":   # 도구 호출 끝 → 탈출
            return assistant_msg["content"][-1]["text"]

        tool_results = [run_tool(b["toolUse"]) for b in assistant_msg["content"] if "toolUse" in b]
        messages.append({"role": "user", "content": tool_results})
        # 다시 루프 → 모델이 또 도구를 쓸 수도 있음

    return "[ max_turns 초과 — 응답 없음 ]"
```

```
ask_with_tools vs run_conversation 차이:

ask_with_tools  → if/else 분기 1회  → 도구 최대 1번
run_conversation → for 루프         → 도구 N번 반복 가능

예: "서울 시간이랑 날씨 둘 다 알려줘"
  → get_time 호출 → 결과 반환
  → get_weather 호출 → 결과 반환
  → 최종 답변 생성
  (2번 연속 도구 호출 → run_conversation만 처리 가능)
```

### 추가된 것 2: `toolChoice` — 도구 선택 방식 제어

```python
# ① auto (기본) — 모델이 알아서 판단
TOOL_CONFIG = {"tools": [WEATHER_TOOL, TIME_TOOL]}
# "서울 날씨 어때?" → 도구 호출
# "안녕하세요"      → 도구 안 씀 (그냥 대답)

# ② any — 무조건 도구 1개 이상 호출 강제
TOOL_CONFIG = {
    "tools": [WEATHER_TOOL, TIME_TOOL],
    "toolChoice": {"any": {}}
}
# "안녕하세요"도 → 억지로 도구 중 하나를 호출함

# ③ tool — 특정 도구만 강제
TOOL_CONFIG = {
    "tools": [WEATHER_TOOL, TIME_TOOL],
    "toolChoice": {"tool": {"name": "get_weather"}}
}
# 반드시 get_weather만 호출, 다른 도구는 무시
```

```
언제 쓰나?
  auto  → 일반적인 챗봇 (대부분의 경우)
  any   → 반드시 외부 데이터가 필요한 서비스 (검색 엔진 등)
  tool  → 특정 도구 테스트 / 파이프라인 고정 호출
```

## 5단계 — P5_06: `chat_turn()` — 대화 히스토리 유지

**목적:** 여러 턴(turn)에 걸친 대화에서 이전 내용을 기억하면서 도구도 사용합니다.

### 추가된 것: 외부 messages 리스트 공유

```python
# messages를 함수 밖에서 관리 → 여러 턴에 걸쳐 누적
conversation_history = []

def chat_turn(user_text: str) -> str:
    conversation_history.append({"role": "user", "content": [{"text": user_text}]})
    return run_conversation(conversation_history)   # 기존 히스토리 포함해서 호출

# 사용
chat_turn("서울 날씨 알려줘")       # get_weather 호출
chat_turn("거기 지금 몇 시야?")     # "거기" = 서울 → 이전 대화 기억해서 get_time 호출
```

```
messages 내부 구조 (히스토리에 쌓이는 것들):
  user      → 사용자 질문
  assistant → 모델 응답 (toolUse 블록 포함 가능)
  user      → toolResult (도구 실행 결과)
  assistant → 최종 답변
  user      → 다음 질문
  ...
  (toolUse/toolResult도 히스토리에 포함되어야 문맥 유지)
```

## 최종 — P6_08: `@tool` 데코레이터 — 모든 것이 자동

**목적:** P5에서 직접 작성하던 모든 것을 프레임워크(Strands)가 처리합니다.

### P5 vs P6 코드 비교

```python
# ━━━━━ P5 방식 (수동) ━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
def get_weather(city: str) -> dict:         # ① 함수 구현
    ...

WEATHER_TOOL = {                            # ② toolSpec 수동 작성
    "toolSpec": {
        "name": "get_weather",
        "description": "...",
        "inputSchema": {"json": {...}},
    }
}
TOOL_FUNCTIONS = {"get_weather": get_weather}  # ③ 등록표 수동 관리
TOOL_CONFIG = {"tools": [WEATHER_TOOL]}        # ④ toolConfig 조립

def run_tool(tool_use): ...                 # ⑤ toolResult 조립 함수
def run_conversation(messages): ...        # ⑥ 반복 루프 함수

answer = run_conversation([{"role":"user","content":[{"text":"서울 날씨"}]}])


# ━━━━━ P6 방식 (@tool + Agent) ━━━━━━━━━━━━━━━━━━━━━━━━
@tool
def weather_forecast(city: str) -> str:    # ① 함수 구현 + @tool 한 줄
    """지정한 도시의 현재 날씨를 조회한다.
    Args:
        city: 도시 이름 (영문). 예: Seoul  ← docstring이 toolSpec 대체
    """
    ...

agent = Agent(tools=[weather_forecast])    # ② 등록
answer = agent("서울 날씨 알려줘")         # ③ 끝 — 나머지는 Agent가 알아서
```

### 무엇이 자동화됐나

```
P5에서 개발자가 하던 일        P6에서 누가 하나
─────────────────────────────────────────────────
② toolSpec JSON 작성          → @tool + docstring 분석으로 자동 생성
③ TOOL_FUNCTIONS 등록표 관리  → Agent(tools=[...]) 한 줄로 대체
④ toolConfig 조립             → Agent 내부에서 자동 처리
⑤ run_tool() — toolResult 조립→ Agent 내부에서 자동 처리
⑥ run_conversation() 루프     → Agent 내부에서 자동 처리
   대화 히스토리 관리           → Agent 내부에서 자동 처리
```

## 학습 포인트 정리

```
1. P5를 먼저 배우는 이유
   → @tool 한 줄이 얼마나 많은 일을 대신하는지 실감하기 위해

2. toolUseId가 핵심
   → Step 2에서 받은 ID를 Step 3 toolResult에 그대로 돌려줘야 함
   → 불일치 시 ValidationException 발생

3. roles must alternate 규칙
   → user → assistant → user(toolResult) 순서 반드시 지켜야 함

4. description이 도구 호출 여부를 결정
   → 설명이 불명확하면 모델이 도구를 안 쓰고 그냥 답해버림
   → P6의 @tool은 docstring의 Args 설명이 이 역할을 함

5. run_conversation의 max_turns
   → 무한 루프 방지 안전장치
   → 실서비스에서는 5~10이 적당
```
