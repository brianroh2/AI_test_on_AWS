# P6 — Strands Agent 코드 진화 가이드

이 문서는 P6 시리즈의 코드가 파일을 거치면서 어떻게 진화하는지,  
그 흐름을 단계별로 설명합니다.  
P5에서 직접 만들던 Tool Use 반복 루프가 Strands Agent 안으로 들어가면서  
개발자가 집중해야 할 것이 "어떻게 호출하나"에서 "무엇을 할 수 있나"로 바뀝니다.

---

## P5 → P6 전환 핵심

```
P5: 개발자가 toolSpec, run_tool, run_conversation, 히스토리를 직접 작성
         ↓
P6: Strands Agent가 이 모든 것을 내부에서 처리
    개발자는 @tool 함수 + Agent() 생성만 하면 됨
```

---

## 전체 진화 한눈에 보기

```
P6_01  →  P6_02  →  P6_03  →  P6_04  →  P6_05
환경확인   Agent기본  시스템프롬  대화메모리  세션영속성

  →  P6_06  →  P6_07  →  P6_08  →  P6_09  →  P6_10/11
     실행통계   내장도구   커스텀도구   MCP연동    ReAct/ReWOO
     (참고)    (가져다씀)  (직접만듦)  (외부서버)  (패턴비교)
```

## 1단계 — P6_01/02: Agent 기본 사용법

**목적:** Strands Agent를 처음 만들고 응답을 받는 최소 구조를 이해합니다.

### 최소 코드

```python
from strands import Agent
from strands.models import BedrockModel
import boto3

model = BedrockModel(
    model_id="global.anthropic.claude-sonnet-4-6",
    boto_session=boto3.Session(region_name="us-east-1"),
    streaming=True,
)

agent = Agent(model=model)
agent("인공지능이란 무엇인가요?")   # 끝 — 이게 전부
```

### 출력 방식 3가지 (P6_02)

```
┌─────────────────────────────────────────────────────────┐
│  streaming=True  + 기본 handler  → 토큰 단위 실시간 출력 │
│  streaming=False + 기본 handler  → 완성 후 한 번에 출력  │
│  callback_handler=None           → 출력 억제             │
│                    ↑ 이때 result = agent("...") 로 캡처  │
└─────────────────────────────────────────────────────────┘
```

```python
# 출력 억제 + 결과 캡처 패턴 (Streamlit, API 서버에서 필수)
agent = Agent(model=model, callback_handler=None)
result = agent("질문")
text = str(result)   # 문자열로 변환
```

> **P5 대비:** P5에서는 `converse()` → `resp["output"]["message"]["content"][0]["text"]`  
> 로 꺼내야 했지만, P6는 `str(result)` 한 줄로 끝납니다.

## 2단계 — P6_03: System Prompt

**목적:** Agent의 역할과 행동 방식을 system_prompt로 고정합니다.

```python
agent = Agent(
    model=model,
    system_prompt="""당신은 바리스타 AI입니다.
    - 커피 관련 질문에만 답합니다.
    - 항상 친근하고 따뜻한 어조를 사용합니다.
    - 다른 주제는 정중히 거절합니다.
    """,
)
```

```
system_prompt의 역할:
  → 모델의 정체성, 말투, 행동 범위를 고정
  → 도구 사용 방식도 여기서 지시 가능
     예: "날씨 질문이면 반드시 weather_forecast 도구를 사용하라"
  → P7 JARVIS에서 "3문장 이내로 답변" 지시에 활용
```

## 3단계 — P6_04/05: 메모리와 세션

**목적:** 대화 기록을 유지하고(메모리), 프로그램을 껐다가 켜도 이전 대화를 복원합니다(세션).

### 메모리 (P6_04) — Agent가 자동 관리

```python
agent = Agent(model=model)

agent("내 이름은 김철수야.")
agent("내 이름이 뭐라고 했지?")   # "김철수"라고 기억하고 답변

# 히스토리 확인
print(agent.messages)   # [{"role":"user",...}, {"role":"assistant",...}, ...]
```

```
┌─────────────────────────────────────────────────────────┐
│  P5_06의 chat_turn()에서 개발자가 직접 관리하던          │
│  conversation_history 리스트를                          │
│  Strands Agent가 내부의 agent.messages로 자동 관리       │
└─────────────────────────────────────────────────────────┘
```

### 세션 (P6_05) — 대화 영속성

```python
# 1회차: 대화 후 저장
agent = Agent(model=model)
agent("나는 파이썬 개발자야.")
save_session("user_001", agent.messages)   # sessions/user_001.json 저장

# 2회차: 불러와서 이어서 대화
messages = load_session("user_001")       # JSON 로드
agent2 = Agent(model=model, messages=messages)   # 히스토리 주입
agent2("내가 뭐 하는 사람이라고 했지?")   # "파이썬 개발자"라고 기억
```

```
메모리 vs 세션 차이:
  메모리 → 프로그램 실행 중 유지 (종료하면 사라짐)
  세션   → 파일/DB에 저장 → 프로그램 재시작 후에도 복원
```

## 4단계 — P6_07/08: 도구 사용 ★ 핵심

**목적:** Agent에 도구를 붙여 외부 정보를 가져오거나 계산합니다.  
P5에서 직접 작성하던 toolSpec/run_tool/루프가 모두 자동화됩니다.

### 내장 도구 (P6_07) — import 해서 바로 사용

```python
from strands_tools import calculator, current_time, file_read, file_write, http_request

agent = Agent(
    model=model,
    tools=[calculator, current_time, file_write, file_read],
)

agent("1234 × 5678 계산해줘")          # calculator 자동 호출
agent("지금 서울 시간 알려줘")          # current_time 자동 호출
agent("result.txt에 '완료' 라고 써줘")  # file_write 자동 호출
```

```
내장 도구 목록:
  calculator    → 수식 계산
  current_time  → 현재 시간 (타임존 지원)
  file_read     → 파일 읽기
  file_write    → 파일 쓰기
  http_request  → HTTP GET/POST (BYPASS_TOOL_CONSENT=true 필요)
```

### 커스텀 도구 (P6_08) — @tool 데코레이터로 직접 제작

```python
from strands import Agent, tool

@tool
def weather_forecast(city: str) -> str:
    """지정한 도시의 현재 날씨를 조회한다.

    Args:
        city: 날씨를 조회할 도시 이름 (영문). 예: Seoul, Tokyo
    """          ↑ docstring의 Args 설명이 P5의 toolSpec inputSchema를 대체
    resp = requests.get(f"https://wttr.in/{city}?format=j1", timeout=5)
    cc   = resp.json()["current_condition"][0]
    return f"{city}: {cc['weatherDesc'][0]['value']}, {cc['temp_C']}°C"

agent = Agent(model=model, tools=[weather_forecast])
agent("서울 날씨 알려줘")   # weather_forecast 자동 호출
```

### P5 vs P6_08 도구 등록 비교

```
P5 방식 (수동):                        P6 방식 (@tool):
─────────────────────────────────────────────────────
WEATHER_TOOL = {                       @tool
  "toolSpec": {                        def weather_forecast(city: str) -> str:
    "name": "get_weather",                 """...
    "description": "...",                  Args:
    "inputSchema": {                           city: ...
      "json": {                           """
        "type": "object",                  ...
        "properties": {
          "city": {"type":"string"}   ← docstring이 이 역할
        },
        "required": ["city"]
      }
    }
  }
}
TOOL_FUNCTIONS = {"get_weather": fn}   agent = Agent(tools=[weather_forecast])
TOOL_CONFIG = {"tools": [WEATHER_TOOL]}
```

## 5단계 — P6_09: MCP — 외부 도구 서버 연결

**목적:** 내 코드 안에 함수를 만드는 것이 아니라,  
**별도 서버**가 제공하는 도구를 Agent에 연결합니다.

```
┌────────────────────────────────────────────────────────┐
│  기존 (@tool):  Agent ←→ 내 코드 안의 함수             │
│  MCP:           Agent ←→ [MCP 프로토콜] ←→ 외부 서버   │
│                                                        │
│  외부 서버가 도구를 제공하면 Agent는 그걸 자기 도구처럼  │
│  사용 — 서버가 어떤 언어로 만들어졌는지 상관없음         │
└────────────────────────────────────────────────────────┘
```

```python
from mcp import StdioServerParameters
from strands.tools.mcp import MCPClient

mcp_client = MCPClient(lambda: StdioServerParameters(
    command="python3",
    args=["woodcutter_mcp_server.py"],   # 외부 MCP 서버
))

with mcp_client:                          # with 블록 안에서만 사용 가능
    tools = mcp_client.list_tools_sync()  # 서버가 제공하는 도구 목록 가져오기
    agent = Agent(model=model, tools=tools)
    agent("참나무 5그루 벌목하고 목재로 가공해줘")
```

```
⚠️ with 블록 필수: with mcp_client: 밖에서 도구 사용 불가
⚠️ 서버 파일 경로: woodcutter_mcp_server.py가 같은 디렉토리에 있어야 함
```

## 6단계 — P6_10/11: ReAct vs ReWOO 패턴 ★ 핵심

**목적:** 도구를 여러 개 쓰는 복잡한 작업을 처리하는 두 가지 전략을 비교합니다.

### ReAct (P6_10) — 생각하고 → 행동하고 → 관찰하고 → 반복

```
┌─────────────────────────────────────────────────────────┐
│  질문: "서울과 런던 날씨 비교 + 두 도시 기온 평균 계산"  │
│                                                         │
│  Thought  → "서울 날씨부터 조회해야겠다"                │
│  Action   → weather_forecast("Seoul")                   │
│  Observe  → "서울: 맑음, 29°C"                          │
│                                                         │
│  Thought  → "이제 런던 날씨 조회"                       │
│  Action   → weather_forecast("London")                  │
│  Observe  → "런던: 흐림, 15°C"                          │
│                                                         │
│  Thought  → "평균 계산해야 함"                          │
│  Action   → calculator("(29+15)/2")                     │
│  Observe  → "22"                                        │
│                                                         │
│  최종 답변 생성                                         │
└─────────────────────────────────────────────────────────┘
```

```python
REACT_PROMPT = """당신은 ReAct 패턴으로 동작하는 AI입니다.
매 단계마다 Thought → Action → Observation 순서로 진행하세요."""

react_agent = Agent(
    model=model,
    tools=[weather_forecast, web_search, calculator],
    system_prompt=REACT_PROMPT,
)
react_agent("서울과 런던 날씨 비교하고 기온 평균 계산해줘")
```

### ReWOO (P6_11) — 계획 먼저 → 실행 → 종합

```
┌─────────────────────────────────────────────────────────┐
│  질문: "서울과 런던 날씨 비교 + 두 도시 기온 평균 계산"  │
│                                                         │
│  [Planner] 전체 계획 수립 (도구 호출 없음)               │
│  → Step1: weather_forecast("Seoul")   → 서울 날씨 조회  │
│  → Step2: weather_forecast("London")  → 런던 날씨 조회  │
│  → Step3: calculator("(29+15)/2")     → 평균 계산       │
│                                                         │
│  [Worker] 계획대로 순서대로 실행                         │
│  → Step1 실행 → Step2 실행 → Step3 실행                  │
│                                                         │
│  [Solver] 모든 결과를 종합해서 최종 답변 생성            │
└─────────────────────────────────────────────────────────┘
```

```python
# Planner: 계획만 세우는 Agent (callback_handler=None 으로 출력 억제)
planner = Agent(model=model, system_prompt=PLANNER_PROMPT, callback_handler=None)
plan_json = str(planner(task))   # JSON 계획표 반환

# Worker: 계획에 따라 도구 순서대로 실행
worker = Agent(model=model, tools=[weather_forecast, calculator], callback_handler=None)
results = []
for step in plan:
    result = worker(f"{step['tool']}({step['input']})")
    results.append(result)

# Solver: 결과 종합 → 최종 답변
solver = Agent(model=model, system_prompt=SOLVER_PROMPT)
solver(f"계획: {plan}\n결과: {results}\n질문: {task}")
```

### ReAct vs ReWOO 언제 쓰나

```
┌──────────────┬──────────────────────┬──────────────────────┐
│ 항목          │ ReAct                │ ReWOO                │
├──────────────┼──────────────────────┼──────────────────────┤
│ 도구 호출 방식│ 결과 보고 다음 결정  │ 계획 후 순서대로 실행 │
│ LLM 호출 횟수│ 도구마다 1회 (많음)  │ Planner+Solver 2회   │
│ 유연성        │ 높음 (중간 방향 변경)│ 낮음 (계획 고정)     │
│ 비용          │ 상대적으로 높음      │ 상대적으로 낮음      │
│ 적합한 상황  │ 불확실·복잡한 작업   │ 명확·예측 가능한 작업 │
└──────────────┴──────────────────────┴──────────────────────┘
```

## 전체 진화 요약표

```
┌──────────┬──────────────────────────┬───────────────────────────────────┐
│ 파일      │ 핵심 패턴                 │ 무엇이 달라졌나                    │
├──────────┼──────────────────────────┼───────────────────────────────────┤
│ P6_01    │ Agent(model=model)       │ P5의 converse() → Agent() 한 줄   │

│ P6_03    │ system_prompt            │ Agent 역할·말투·행동 범위 고정     │
│ P6_04    │ agent.messages (자동)    │ P5_06의 히스토리 관리 자동화       │
│ P6_05    │ save/load session        │ 메모리를 파일로 저장 → 재시작 복원  │
│ P6_06    │ agent 실행 통계          │ 토큰/시간/도구 사용량 모니터링      │
│ P6_07    │ strands_tools 내장 도구  │ import 한 줄로 계산기/시간/파일 사용│
│ P6_08 ★  │ @tool 커스텀 도구        │ P5 toolSpec 80줄 → @tool + 10줄   │
│ P6_09    │ MCP 외부 서버 연결       │ 다른 언어 서버의 도구도 사용 가능   │
│ P6_10 ★  │ ReAct 패턴               │ 생각→행동→관찰 반복 (유연)         │
│ P6_11 ★  │ ReWOO 패턴               │ 계획→실행→종합 분리 (효율)         │
│ P6_10    │ Streamlit 시각화         │ 두 패턴 동일 질문으로 탭 비교       │
│ (Strm)   │                          │                                   │
└──────────┴──────────────────────────┴───────────────────────────────────┘
```

## 학습 포인트 정리

```
1. P5 → P6 가장 큰 변화
   → toolSpec/run_tool/run_conversation 이 Agent 내부로 들어감
   → 개발자는 "무엇을 할 수 있나(@tool)"에만 집중

2. @tool docstring이 toolSpec을 대체
   → name: 함수 이름으로 자동 결정
   → description: docstring 첫 줄
   → inputSchema: Args 섹션 + 타입 힌트로 자동 생성
   → Args 설명이 빈약하면 모델이 파라미터를 잘못 채움

3. callback_handler=None 패턴
   → Streamlit / API 서버에서 Agent 출력을 직접 제어할 때 필수
   → str(result) 로 텍스트 추출

4. ReAct vs ReWOO 선택 기준
   → 작업 경로가 불확실하다 → ReAct
   → 작업 단계가 명확하다 → ReWOO (비용 절감)
   → P7 JARVIS는 ReAct 방식 (사용자 질문이 다양하므로)

5. MCP with 블록 주의
   → with mcp_client: 블록 밖에서 도구 호출 시 연결 끊김 오류
```
